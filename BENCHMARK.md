# Sentinel — Held-Out Benchmark (external)

An independent accuracy check of the `anwen-labs/sentinel` engine against **18 fixtures it
was never tuned on**, run through the real engine (`crates/harness`, release build).
Purpose: produce reproducible evidence to replace the unbacked "zero false positives" /
"precision-recall 1.000" marketing claim with numbers a skeptic can re-run.

**Build note:** built on rustc 1.75 with the `cli` crate (clap-only) dropped from the
workspace; the engine, parsers, rule packs, and harness compile and run unchanged. Findings
were generated by the same engine the CLI and web UI use.

> **Update (2026-06-14) — gaps fixed, re-measured.** The three real misses and the
> precision gap documented below have since been fixed in the engine: the Critical
> `/run/docker.sock` miss, the standalone `aws_vpc_security_group_ingress_rule`, the
> `postgres` weak-default, and the `AKIA…EXAMPLE` comment false positive. Re-running
> this **same 18-fixture held-out set** on the current engine now scores **12 TP /
> 1 FP / 0 FN — precision 0.92, recall 1.00** (the one remaining FP is a correct
> runs-as-root detection the blind fixture under-labels; adjudicated precision 1.00).
> The original point-in-time results below are preserved unedited.

---

## TL;DR

| Corpus | Precision | Recall | Determinism |
|---|---|---|---|
| **Own corpus** (74 fixtures, in-repo) | 1.000 | 1.000 | 74/74 |
| **Held-out** (18 fixtures, this set) — raw harness | 0.727 | 0.667 | 18/18 |
| **Held-out** — after adjudication* | **0.91** | **0.67–0.75** | **18/18** |

\* Adjudication = manual review of every raw FP/FN. Two "FPs" and one "FN" the harness
counted are actually correct detections under a different rule ID than the fixture label
(documented below); they are not engine errors. The reverse never happened.

**The one-line read:** the engine is **sound and high-precision and perfectly
deterministic**, but its rule *coverage* is narrower than "1.000" implies. On realistic,
current config variants it has never seen, it misses a handful of real issues — including
one **Critical**. None of the misses are noise; all three are specific and cheap to fix.
This is exactly the profile you'd expect from a 70-rule, day-one scanner — which is a far
more credible story to a buyer than "zero false positives."

---

## Why this matters

`1.000 / 1.000` on a corpus the author wrote is necessary but not sufficient: the fixtures
and the rules are tuned to each other. The honest question is how the engine does on inputs
it has *not* seen. This set probes that, staying strictly inside Sentinel's stated scope
(the 6 formats, the published rule IDs) — no strawman checks for things it never claimed.

---

## Method

- **11 recall probes** — realistic, *current* variants of in-scope misconfigurations the
  in-repo corpus doesn't cover (e.g. the modern `/run/docker.sock` path, IPv6 `::/0`, the
  standalone `aws_vpc_security_group_ingress_rule` resource, an Ingress-fronted privileged
  pod, newer secret formats).
- **7 precision probes** — hardened/clean configs that a naive scanner over-flags
  (localtime mount, fully-hardened pod, private-CIDR SG, ref-only `.env`, least-privilege
  GHA, relative bind mount).
- Ground-truth labels live in each fixture's `# EXPECT:` header (the harness's own format).
- Scored with the in-repo harness (`cargo run -p eval`), then every FP/FN reviewed by hand.

Reproduce:

```
# drop these 18 files into crates/harness/corpus/ (back up the original first)
cargo run --release -p harness        # binary name: eval
```

## Adjudication

### The 4 raw misses

1. **`r01` — `/run/docker.sock` → DOCKER-SOCKET-MOUNT missed. REAL, CRITICAL.**
   The rule matches the mount source by exact string `/var/run/docker.sock`
   (`pack-sentinel-core/src/lib.rs`, `r1_docker_socket`). On modern systemd hosts
   `/var/run` is a symlink to `/run`, and `/run/docker.sock` is the canonical path used in
   countless real compose files. `/run` is also absent from the sensitive-host-path list,
   so the mount hits **neither** rule and produces **zero findings** — full host root, silently
   passed. *Note the asymmetry:* `r11` mounts the same socket via a K8s `hostPath` and **is**
   caught, because the K8s hostPath rule is path-agnostic. The gap is specific to the
   Compose exact-string match.
   **Fix:** match on the `docker.sock` basename (or normalize `/var/run`→`/run`) instead of
   an exact literal; add `/run` to the sensitive-path prefixes.

2. **`r03` — standalone `aws_vpc_security_group_ingress_rule` with `0.0.0.0/0` → missed. REAL.**
   The parser detects open ingress inside inline `ingress {}` blocks of `aws_security_group`,
   but not the standalone `aws_vpc_security_group_ingress_rule` resource — the pattern AWS
   has recommended since provider v4/v5 and the one most current Terraform actually uses.
   **Fix:** add a parser path for the standalone ingress-rule resource (`cidr_ipv4` /
   `cidr_ipv6` attributes).

3. **`r04` — K8S-REACHABLE-NODE-COMPROMISE missed (Ingress path). REAL.**
   The privileged container itself was caught. But the cross-resource reachability rule — a
   headline capability ("reachable node-compromise chains") — only treats `LoadBalancer` /
   `NodePort` Services as internet-reachable. An `Ingress` → `ClusterIP` Service →
   privileged Deployment is the more common production exposure, and the reachability
   escalation is missed.
   **Fix:** model `Ingress` (and `Gateway`) as external entry points in the reachability graph.

4. **`r09` — WEAK-DEFAULT-CREDENTIAL missed for value `postgres`. MINOR / not silent.**
   The weak-default *value* list doesn't include the literal `postgres`. However the same
   line fired **SECRET-IN-ENVIRONMENT**, so the hardcoded credential was not missed — only
   mislabeled as a generic env-secret rather than a known weak default.
   **Fix:** add common service defaults (`postgres`, `mysql`, `redis`, `guest`) to the
   weak-value list.

### The 3 raw false positives

- **`p15` — SECRET-AWS-ACCESS-KEY on `AKIAIOSFODNN7EXAMPLE`. REAL precision gap (1).**
  This is the canonical AWS *documentation example* access key (the `…EXAMPLE` suffix), a
  deliberate non-secret that precision scanners (gitleaks, trufflehog, detect-secrets)
  allowlist. It fired even though the string is inside a `#` comment, because high-confidence
  token matching intentionally runs on comment lines too (`secrets-parser/src/lib.rs`: the
  `is_comment` guard gates only the *generic* detector, line 104). Scanning comments is a
  defensible call (people comment out real keys); flagging the known docs example key is the
  cheap precision win.
  **Fix:** allowlist the well-known example keys (`AKIAIOSFODNN7EXAMPLE`,
  `AKIAI44QH8DHBEXAMPLE`, `wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY`).

- **`r04` — K8S-CONTAINER-RUNS-AS-ROOT. NOT an FP.** The privileged pod sets no
  `runAsNonRoot`, so it does run as root; the Low finding is correct. Under-labeled by me.

- **`r09` — SECRET-IN-ENVIRONMENT. NOT an FP.** A hardcoded password in an env var *is* a
  secret in the environment; the Medium finding is correct. Under-labeled by me.

### Adjudicated numbers

- **Real false positives: 1** of 11 findings produced (`AKIA…EXAMPLE`). → precision ≈ **0.91**.
- **Real misses: 3** (one Critical, two High) + 1 minor mislabel. → recall **0.67** strict,
  **0.75** crediting the alternate-rule catch on `r09`.
- **Determinism: 18/18** — the reproducibility claim holds on held-out inputs.

## Strengths confirmed (probes that tried to break it and failed)

- **Secret format coverage is broad and current** — ASIA STS keys, GitHub fine-grained PATs
  (`github_pat_`), `gho_/ghu_/ghs_`, Stripe restricted keys (`rk_live_`), all Slack `xox*`
  variants, SendGrid, Google — all caught. Three of my predicted "newer format" gaps were
  non-gaps.
- **IPv6 `::/0`** open security groups: caught.
- **`wget … | bash`**: caught (the curl-pipe rule is broader than its name).
- **High precision on hardened configs** — 6 of 7 clean probes produced zero findings,
  including the localtime-mount and relative-bind-mount cases that commonly cause FPs.
- **Perfect determinism** across every fixture.

---

## Recommended claim language

Drop the absolutes — they're the first thing a security buyer tests, and this benchmark
breaks them:

- ~~"zero false positives"~~ → **"high-precision, low-noise"** (1 known-example FP here is honest).
- ~~"precision / recall 1.000"~~ → **"1.000 on our published corpus; we also run a held-out
  set — see the numbers and known gaps below."**
- Add a `KNOWN-GAPS.md` (the three above) and a public `cargo run -p eval`-able held-out
  corpus. *Publishing your misses is the strongest trust signal you have* — it says you
  measure honestly, which is the entire thesis of the product.
